# Kamailio integration for `action='call_menu'`

Iter 155 introduced a new action shape on `/api/internal/inbound-route`:

```json
{
  "action": "call_menu",
  "call_menu_id": "<uuid>",
  "reason": "did_match_entry_menu",
  "to": "+15551234567",
  "in_group_id": "<uuid>"
}
```

Returned when the matched in-group has `entry_call_menu_id` set. Iter 156
also surfaces this for the campaign no-agent drop (outbound side, handled
by the FreeSWITCH dialplan directly — Kamailio doesn't see those calls).

Kamailio's existing inbound branch needs one more case to consume the
`call_menu` action. Add the snippet below to the route block that
currently switches on `$avp(action)` after the HTTP call.

## Snippet for `kamailio.cfg`

```ini
# ... inside the route that POSTs to /api/internal/inbound-route ...

http_async_query("$var(api_url)", "DECISION_RECEIVED");

route[DECISION_RECEIVED] {
    if ($http_status != 200) {
        send_reply("503", "Service Unavailable");
        exit;
    }

    # Parse JSON body
    jansson_get("action", "$var(reply_body)", "$var(action)");

    # Iter 155 — entry-time call menu (in-group bound)
    # Iter 156 — same shape may be used elsewhere later
    if ($var(action) == "call_menu") {
        jansson_get("call_menu_id", "$var(reply_body)", "$var(menu_id)");
        # Hand the call to FreeSWITCH directly into the
        # iter-152 auto-generated extension call_menu_<id>.
        # The DialerOS FreeSWITCH gateway listens at $sel(cfg_get.dialeros.fs_dispatcher).
        $ru = "sip:call_menu_" + $var(menu_id) + "@" + $sel(cfg_get.dialeros.fs_dispatcher);
        t_relay();
        exit;
    }

    # Existing actions (forward / queue / reject) handled below as before.
    if ($var(action) == "forward") {
        # ...
    }
    if ($var(action) == "queue") {
        # send caller to FS dialeros-inbound-queue extension
        $ru = "sip:dialeros-inbound-queue@" + $sel(cfg_get.dialeros.fs_dispatcher);
        t_relay();
        exit;
    }
    if ($var(action) == "reject") {
        send_reply("403", "Rejected");
        exit;
    }
}
```

The dispatcher target (`cfg_get.dialeros.fs_dispatcher`) should already
be configured for the existing queue/forward paths — same FreeSWITCH
instance, just a different extension name in the URI user part.

## Smoke test

```bash
TOKEN=$(grep '^KAMAILIO_INBOUND_TOKEN=' /etc/dialeros/admin.env | cut -d= -f2-)
curl -fsS -X POST \
  -H "Content-Type: application/json" \
  -H "X-Inbound-Token: ${TOKEN}" \
  -d '{"to": "+15551234567", "from": "+15557654321", "call_id": "test-1"}' \
  http://127.0.0.1:1111/api/internal/inbound-route
```

If the DID `+15551234567` belongs to an in-group with `entry_call_menu_id`
set, the response is `{"action": "call_menu", "call_menu_id": "<uuid>", ...}`.

## Operator setup

1. Create a Call Menu in the admin GUI (`/call-menus/add`).
2. Open the in-group's edit page (`/in-groups/<id>/edit`).
3. In the "Call menu wiring" section at the bottom, pick the menu in
   "Entry menu" → Save.
4. Apply the Kamailio snippet above + `kamcmd cfg.reload` (or restart
   kamailio).
5. Place a test inbound to the DID. Kamailio's HTTP call resolves the
   action; FreeSWITCH's `call_menu_<id>.xml` (auto-generated by iter 152)
   plays the prompt + handles the DTMF branch.

If you don't see the call_menu action being chosen, double-check:
- The DID is mapped to the in-group (`/dids/<did>`).
- The in-group has `entry_call_menu_id` set (`pragma table_info(in_groups);`
  + `select * from in_groups where id=...;`).
- `/etc/freeswitch/dialplan/default/call_menu_<id>.xml` exists on disk
  (iter 152 writes it on every save).
- `fs_cli -x reloadxml` returned `+OK [Success]` at last admin restart.
